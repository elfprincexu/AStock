## 1. 项目概述

### 一句话描述

> AStock 是一个面向中国 A 股市场的全栈量化数据平台，集成 6 大数据源、10 个功能模块，覆盖从数据采集、行情展示、智能选股、量化分析、AI 诊断到策略交易的完整投研闭环。

### 30 秒电梯演讲

AStock 是我独立设计开发的 A 股全栈量化平台。后端用 Python FastAPI 构建异步 API，前端用 Vue 3 + Element Plus + ECharts 做数据可视化，PostgreSQL 存储约 1000 万条 K 线数据，Redis + Celery 处理定时任务和异步调度。

系统最核心的设计是**多数据源聚合引擎**——聚合了新浪、腾讯、东方财富、akshare、tushare、baostock 共 6 个数据源，通过优先级 fallback 机制确保单一源故障时自动切换，数据可用性接近 100%。

在此基础上，我实现了四个高价值模块：**50 因子跨截面量化选股引擎**（含 IC 验证、LightGBM ML 增强、Walk-forward 回测）、**LLM 驱动的 AI 智能诊断**（支持 Gemini/OpenAI/DeepSeek 等多模型）、**条件单策略交易引擎**（支持模拟交易和平安证券 QMT 实盘）、以及**智能选股系统**（含量价模式识别）。

整个系统代码约 2.2 万行（后端 1.3 万 + 前端 7900 + 测试 1800），覆盖了全栈开发、数据工程、量化金融、AI 集成等多个领域。

## 2. 项目初衷与动机

### 为什么做这个项目？

1. **实际投研需求**：个人对 A 股量化投资有浓厚兴趣，市面上的免费工具要么数据源单一不稳定，要么只有数据没有分析能力，要么缺乏交易闭环。我希望构建一个从数据采集到策略执行的完整系统。

2. **全栈能力展示**：项目天然需要前后端分离、数据库设计、任务调度、第三方 API 集成、数据可视化等多种技术栈的协同，是一个极好的全栈工程能力载体。

3. **解决真实痛点**：
   - A 股数据源不稳定（新浪经常限流、东方财富有 IP 限制）→ 多源 fallback 机制
   - 散户缺乏量化分析工具 → 50 因子选股引擎 + 回测系统
   - AI 分析报告质量参差不齐 → 技术指标预计算 + 结构化 prompt + LLM 输出验证
   - 条件单执行需要盯盘 → 自动化策略引擎 + 5 秒级行情监控

### 项目定位

这不是一个"学习项目"或"玩具项目"，而是一个**可以实际使用的生产级系统**。它已经在实际投研中运行，管理着 5,684 只 A 股的 10 年历史数据（约 1020 万条 K 线记录），并支持日常自动更新和交易监控。

### 架构分层

| 层级 | 职责 | 技术选型 |
|------|------|---------|
| **表示层** | 用户界面、数据可视化、交互 | Vue 3 + Element Plus + ECharts |
| **API 层** | RESTful 接口、认证、权限 | FastAPI + Pydantic + JWT |
| **业务逻辑层** | 数据聚合、量化计算、AI分析、交易引擎 | Python Services |
| **数据访问层** | ORM 映射、数据库操作 | SQLAlchemy 2.0 (async) |
| **基础设施层** | 数据库、缓存、任务队列、容器 | PostgreSQL + Redis + Celery + Docker |

### 为什么选择这个架构？

- **前后端分离**：前端可独立迭代，后端 API 可被其他客户端复用
- **异步优先**：FastAPI + asyncpg 充分利用 Python 异步能力，I/O 密集型场景（6 个数据源并发请求）性能优异
- **任务队列解耦**：耗时操作（全量数据更新、策略监控、量化分析）通过 Celery 异步执行，不阻塞 API 响应
- **Docker 容器化**：PostgreSQL、Redis、Grafana 一键启动，环境一致性有保障

## 6. 核心模块实现详解

### 6.1 多数据源聚合引擎 (StockDataAggregator)

**这是系统最核心的基础设施，也是最值得讲的技术点。**

#### 设计思路

A 股免费数据源都有各自的问题：新浪限流、东方财富 IP 封禁、akshare 同步阻塞慢、tushare 有调用限额。单一数据源无法保证可用性。我设计了一个聚合层，统一抽象 6 个数据源，通过优先级 fallback 确保数据获取的可靠性。

#### 核心设计

```
1. 抽象基类 DataSourceClient
   ├── get_realtime_quote(code, market)
   ├── get_daily_klines(code, market, days)
   └── search_stock(keyword)

2. 6 个具体实现
   ├── AkShareClient
   ├── TushareClient
   ├── BaostockClient
   ├── EastMoneyClient
   ├── SinaClient
   └── TencentClient

3. StockDataAggregator (聚合器)
   ├── 配置优先级列表 (DATA_SOURCE_PRIORITY)
   ├── 按数据类型重排序
   │   ├── 实时行情: sina → tencent → eastmoney → akshare (延迟优先)
   │   └── 日 K 线: akshare → tushare → baostock → eastmoney (完整性优先)
   ├── 逐源尝试，失败自动切换下一源
   └── 两阶段 K 线获取 (完整源 → 不完整源兜底)
```

#### 关键技术细节

- **K 线数据完整性分级**：akshare/tushare/baostock/eastmoney 提供完整 K 线（含成交额和换手率），sina/tencent 只提供 OHLCV（缺少成交额和换手率）。系统将不完整源作为最后兜底，避免数据质量下降。
- **编码兼容**：新浪用 GB18030、腾讯用 GBK，统一解码后输出 UTF-8 标准格式。
- **同步/异步双接口**：FastAPI 路由使用 async 版本，Celery Worker 使用 sync 版本（通过 ThreadPoolExecutor 包装）。
- **缺口感知 K 线补全**：计算数据库中最新 K 线日期与最新预期交易日的差距，只请求缺失天数的数据（`min(gap_days + 10, 500)`），避免重复请求。

#### 面试亮点话术

> "我设计了一个适配器模式 + 策略模式的多源聚合架构。6 个数据源实现统一的抽象接口，聚合器根据数据类型动态调整优先级——实时行情优先选低延迟的 HTTP 源（新浪/腾讯），K 线优先选数据完整的库（akshare/tushare）。当某个源故障时自动 fallback 到下一个，对上层业务完全透明。这个设计让系统在新浪限流、东方财富 IP 封禁等常见故障场景下依然能稳定获取数据。"

### 6.2 量化多因子选股引擎 (QuantEngine)

**这是系统中技术含量最高的模块，约 1400 行代码。**

#### 整体流程

```
输入参数 (预设策略/自定义筛选)
    ↓
1. Universe 构建 (数据库预筛选 → 5000+ 只缩至 300-1500 只)
    ↓
2. 数据准备 (从 daily_klines 加载日 K 线矩阵)
    ↓
3. 因子计算 (50 个因子, 8 大类)
    ↓
4. 截面排名 (每日对所有股票排名)
    ↓
5. IC 分析 (Spearman Rank IC → 因子有效性验证)
    ↓
6. 因子加权 (IC-Decay 指数衰减加权)
    ↓
7. ML 增强 (LightGBM 分类器, 可选)
    ↓
8. 综合信号 (IC 加权 + ML 预测混合, 0-100 分)
    ↓
9. 组合构建 (Buffer Zone 机制, Top-N)
    ↓
10. Walk-forward 回测 (T+1 执行、A 股交易成本)
    ↓
输出: 组合持仓 + 回测绩效 + 因子分析
```

#### 50 个因子（8 大类）

| 类别 | 数量 | 代表因子 |
|------|------|---------|
| **价值** | 3 | EP (1/PE), BP (1/PB), 负对数市值 |
| **动量** | 6 | 5/10/20/60/120 日收益率, 量价加权收益 |
| **反转** | 4 | 5/20 日反转, 隔夜/日内收益 |
| **量能** | 7 | 量比(5/20, 5/60), Amihud 非流动性, 换手率, 量价相关 |
| **波动** | 6 | 20/60 日已实现波动率, 特质波动率, 最大回撤, 下行波动 |
| **技术** | 10 | RSI(14/6), 布林带, 价格位置, MACD, 乖离率, 均线交叉 |
| **资金流** | 7 | VWAP 偏离, 上下影线比, 上涨量占比, 净买入, 主力资金流(Alpha101 风格) |
| **统计** | 7 | 偏度, 峰度, 自相关, 高低相关, 收益-量排名相关 |

#### 因子有效性验证（三重条件）

一个因子被判定为有效，需同时满足：
1. `|IC_mean| >= 0.03` — 有意义的预测信号（学术标准：>=0.03 有意义，>=0.05 强信号）
2. `|IC_IR| >= 0.3` — 足够的预测稳定性（IC_IR = IC_mean / IC_std）
3. `max(ic_pos_rate, 1 - ic_pos_rate) >= 0.55` — 方向一致性（至少 55% 的截面方向一致）

#### 回测参数（模拟 A 股真实环境）

- **T+1 执行延迟**：信号日下单，次日开盘执行（A 股规则）
- **交易成本**：买入 7.6bps (佣金 + 过户费)，卖出 13.2bps (含印花税)
- **Buffer Zone**：新股需排名 <= 25 才入选，持仓股排名 > 40 才被剔除（减少换手）
- **10 日再平衡**：每 10 个交易日调仓一次

#### 自动迭代

系统支持"自动迭代"功能：保存的量化分析结果可以按原始调仓频率定期重新运行，自动计算持仓期间收益、生成新组合、记录买卖变动，实现 Walk-forward 实盘跟踪。

#### 面试亮点话术

> "量化引擎的设计参考了 Qlib 和 Alpha101 的理念，采用跨截面分析方法——每日对所有股票横向排名，而不是单只股票的时序分析。我实现了 50 个因子覆盖价值、动量、反转、量能、波动、技术、资金流、统计 8 个维度，通过 Spearman Rank IC 验证因子有效性，用 IC-Decay 指数衰减加权组合信号，可选 LightGBM 做 ML 增强。回测严格模拟 A 股规则——T+1 执行延迟、买入 7.6bps/卖出 13.2bps 真实交易成本、Buffer Zone 减少换手。"

### 6.3 AI 智能诊断 (AI Analysis)

**约 1400 行代码，将技术分析与 LLM 结合。**

#### 分析流水线

```
1. 加载 60+ 天日 K 线数据 (从数据库)
    ↓
2. 计算技术指标
   - MA5/10/20/60 均线排列
   - MACD (DIF, DEA, BAR)
   - RSI (6/12/24 周期)
   - 乖离率 (BIAS)
   - 支撑位/压力位
   - 量能分析
    ↓
3. 获取实时行情 (通过聚合器, 可选)
    ↓
4. 构建结构化 Markdown Prompt
   - 股票基本信息 + 技术指标数值
   - 投资策略系统 Prompt (风险控制约束)
    ↓
5. 调用 LLM (litellm 统一接口)
    ↓
6. 解析/验证 JSON 输出 (Pydantic Schema)
    ↓
7. 持久化到 analysis_history 表
```

#### 技术亮点

- **预计算指标，不依赖 LLM 做数学**：先用 Pandas/NumPy 精确计算所有技术指标，再以结构化数据喂给 LLM，避免 LLM 数学不准确的问题。
- **多模型支持**：通过 litellm 抽象层，一套代码支持 Gemini、OpenAI、Anthropic、DeepSeek、通义千问、月之暗面等 10+ 家 LLM 服务商。
- **异步任务管理**：分析任务在后台线程池执行，前端通过轮询 (2-3 秒) 获取状态，不阻塞 API。
- **结构化输出验证**：LLM 输出必须符合 Pydantic Schema（含情绪分数、买卖点位、趋势预测等），不合规则重试或标记失败。

### 6.4 策略交易引擎 (Trade Engine)

**条件单执行引擎，支持模拟和实盘两种模式。**

#### 核心概念

```
Strategy (策略)
  ├── Step 1: 低位买入
  │   ├── Condition: price <= 10.50 AND fall_pct >= 2.0
  │   └── Action: BUY 1000 shares @ limit 10.50
  ├── Step 2: 止盈卖出
  │   ├── Condition: profit_pct >= 5.0 OR loss_pct >= 3.0
  │   └── Action: SELL 1000 shares @ market
  └── Steps 按顺序执行, Step 1 完成后 Step 2 激活
```

#### 状态机设计

```
Strategy: draft → active → completed
                ↕ pause    ↕ error
          cancelled ←──────┘

Step: waiting → watching → triggered → filled
                                   ↕ failed
```

#### 执行流程

1. Celery Beat 每 5 秒触发 `trade_monitor_tick`
2. 遍历所有 active 策略，获取实时行情
3. 评估当前激活 Step 的所有 Condition（支持 AND/OR 逻辑）
4. 条件满足 → 执行下单（模拟: 直接更新 cash/holdings/avg_cost；实盘: 通过 xtquant 提交到平安证券 QMT）
5. 当前 Step 完成 → 激活下一个 Step
6. 所有 Step 完成 → 策略标记为 completed
7. 全程记录 TradeExecution 事件日志

#### 实盘交易（平安证券 QMT）

通过 Broker Adapter 模式抽象券商接口：
- `BaseBroker` 抽象基类定义统一接口（下单、撤单、查询持仓/资金）
- `PinganBroker` 实现平安证券 QMT 适配器（通过 xtquant SDK）
- 模拟/实盘切换只需改 strategy.mode 字段

### 6.5 智能选股引擎 (Screener)

#### 三层筛选

1. **基本面筛选**：行业、市值区间、PE 区间、PB 区间
2. **技术面筛选**：价格百分位（250 日高低位）、放量突破（5 日均量 vs 60 日均量）、缩量蓄势
3. **量价模式识别**：「放量上涨 → 缩量回调」复合模式检测

#### 量价模式检测算法

```
1. 回看 4 个月数据, 按周聚合
2. 寻找连续 ≥3 周的放量期 (周成交量 > 基准的 1.4 倍)
3. 验证放量期累计涨幅 10-30%
4. 验证放量后有 ≥10% 回撤, ≥2 周
5. 验证近期量能收缩 (近 5 日总量 < 回看期最低周量 × 1.1)
6. 综合评分 0-100
```

#### 12 个预设策略

涵盖大盘价值股、中盘成长股、底部区域股、放量突破、白酒板块、银行金融、科技硬件、新能源、医药生物等常见投研场景。

## 9. 技术难点与解决方案

### 难点 1：数据源不稳定

**问题**：新浪财经频繁限流（HTTP 403）、东方财富对部分 IP 段封禁、akshare 同步调用导致事件循环阻塞。

**解决方案**：
- 设计多源 fallback 机制，优先尝试轻量 HTTP 源（新浪/腾讯），失败自动切换
- akshare 同步调用通过 `ThreadPoolExecutor` 包装为异步，避免阻塞事件循环
- 按数据类型区分优先级——实时行情追求低延迟，K 线追求数据完整性
- 缺口感知 K 线补全，只请求缺失天数的数据，减少不必要的网络请求

### 难点 2：大规模因子计算性能

**问题**：5000+ 只股票 × 500+ 交易日 × 50 个因子，数据矩阵巨大，单次分析可能耗时数分钟。

**解决方案**：
- 数据库预筛选（Universe Pre-filtering）将股票池从 5000+ 缩减到 300-1500
- NumPy/Pandas 向量化计算，避免 Python 循环
- 异步任务模型：分析在后台线程池执行，前端轮询状态，不阻塞 API
- 因子计算结果缓存到 `quant_factor_daily` 表，避免重复计算

### 难点 3：Celery 异步/同步混合

**问题**：FastAPI 使用 asyncpg（异步），Celery Worker 运行在普通线程中（同步），不能共享 async session。

**解决方案**：
- 后端使用两个数据库连接 URL：`DATABASE_URL`（asyncpg，给 FastAPI）和 `DATABASE_URL_SYNC`（psycopg2，给 Celery）
- 数据聚合器提供 `get_realtime_quote()` (async) 和 `get_realtime_quote_sync()` (sync) 双版本接口
- 统一抓取逻辑封装在 `_full_fetch_one_stock_sync()` 中，手动和定时抓取共享同一套代码

### 难点 4：中文编码兼容

**问题**：新浪 API 返回 GB18030 编码，腾讯 API 返回 GBK 编码，且部分接口使用 HTTP（非 HTTPS）。

**解决方案**：
- 每个数据源客户端内部处理编码转换，统一输出 UTF-8
- 使用 `response.content.decode('gb18030')` / `decode('gbk')` 手动解码
- 对 HTTP 接口（腾讯 `qt.gtimg.cn`）允许非加密连接

### 难点 5：LLM 输出结构化

**问题**：LLM 返回的 JSON 格式不总是合规，可能缺字段、格式错误、包含 markdown 代码块标记。

**解决方案**：
- 使用 `json-repair` 库修复常见的 JSON 格式问题
- Pydantic Schema 严格验证输出结构
- 解析失败时自动重试（最多 2 次）
- 最终仍失败则记录原始输出，标记 status=error

### 难点 6：北交所 (BJ) 兼容性

**问题**：各数据源对北交所股票的支持不一致——新浪不支持、akshare K 线不支持、东方财富实时行情会重定向到延迟端点。

**解决方案**：
- 为每个数据源明确标注 BJ 支持情况
- BJ 实时行情主要走腾讯（`bj{code}` 前缀格式）
- BJ K 线主要走东方财富（secid `0.{code}` 格式）
- BJ 股票识别：代码以 4/8/92 开头

## 13. 高频面试问答 (Q&A)

### Q1: 请简单介绍一下这个项目

> AStock 是我独立设计开发的 A 股全栈量化数据平台。它聚合了新浪、腾讯、东方财富等 6 个数据源，通过优先级 fallback 机制保证数据可用性；内置 50 因子量化选股引擎、LLM 驱动的 AI 诊断、条件单策略交易引擎等 10 个模块。技术上使用 FastAPI + Vue 3 + PostgreSQL + Celery，管理着近 5700 只 A 股、1000 多万条 K 线数据。整个系统代码约 2.2 万行，从数据采集到策略执行形成完整闭环。

### Q2: 为什么选 FastAPI 而不是 Django/Flask？

> 三个原因：第一，FastAPI 原生支持 async/await，我的系统需要同时请求 6 个数据源，异步 I/O 能显著提升并发性能；第二，FastAPI 内置 Pydantic 数据验证，前后端契约清晰，自动生成 Swagger 文档便于调试；第三，项目是纯 API 后端（前端独立），不需要 Django 的模板渲染和 admin 等重量级功能。

### Q3: 多数据源 fallback 是怎么实现的？

> 我设计了一个聚合器架构。首先定义抽象基类 `DataSourceClient`，6 个数据源各自实现 `get_realtime_quote` 和 `get_daily_klines` 方法。`StockDataAggregator` 聚合器维护一个优先级列表，按数据类型动态调整顺序——实时行情优先选低延迟的 HTTP 源（新浪/腾讯），K 线优先选数据完整的源（akshare/tushare）。调用时逐源尝试，某源超时或报错则自动切换到下一个，对上层业务透明。K 线获取还有两阶段机制：先尝试完整源（含成交额和换手率），全失败才用不完整源兜底。

### Q4: 量化引擎的 IC 分析是什么？为什么用 Spearman Rank IC？

> IC (Information Coefficient) 是衡量因子预测能力的核心指标——它衡量因子值和未来收益之间的相关性。我用 Spearman Rank IC（排名相关系数）而不是 Pearson IC，因为：(1) 排名相关对异常值不敏感，金融数据有很多极端值；(2) 我们关心的是排名预测能力（选出排名前 30 的股票），而不是精确的收益预测。一个因子要通过三重验证——IC 均值 >= 0.03（有预测信号）、IC_IR >= 0.3（预测稳定）、方向一致性 >= 55%——才会被纳入最终模型。

### Q5: 前端 7500+ 行全在一个文件里，为什么不拆分？

> 这确实是当前的一个技术债。项目在快速迭代阶段采用了单文件方式以减少模块间的状态传递复杂度，所有状态集中管理避免了 prop drilling 的问题。但我在未来改进规划中已经列入了组件拆分——计划引入 Vue Router 做页面路由、Pinia 做状态管理，将每个功能区拆分为独立组件。这是一个典型的"先跑通再优化"的工程权衡。

### Q6: Celery 和 FastAPI 怎么协作的？有什么坑？

> 最大的坑是异步/同步兼容。FastAPI 使用 asyncpg（异步 PostgreSQL 驱动），Celery Worker 运行在普通线程中，不能用 async session。我的解决方案是配置两个数据库连接——`DATABASE_URL` 用 asyncpg 给 FastAPI，`DATABASE_URL_SYNC` 用 psycopg2 给 Celery。数据聚合器也提供了 async 和 sync 两套接口。此外 Windows 上 Celery Worker 必须用 `--pool=solo` 因为 Windows 缺少 `os.fork()`，这个我也在文档里做了说明。

### Q7: 策略交易的状态机是怎么设计的？

> 策略有两层状态机。策略级别：draft → active → completed（正常流程），也可以随时 pause/cancel。步骤级别：waiting → watching → triggered → filled（或 failed）。步骤按顺序执行——Step 1 filled 后，Step 2 才从 waiting 变为 watching。Celery Beat 每 5 秒触发一次 tick，遍历所有 active 策略，获取实时行情，评估当前激活步骤的条件（支持 AND/OR 组合），条件满足则执行下单。整个过程通过 TradeExecution 事件日志完整记录。

### Q8: 交易成本 7.6bps 和 13.2bps 是怎么算的？

> 这是模拟 A 股真实交易成本。买入 7.6bps = 万 3 佣金 (3bps) + 万 0.2 过户费 (2bps) + 安全余量；卖出 13.2bps = 万 3 佣金 (3bps) + 千 1 印花税 (10bps) + 过户费 + 余量。印花税是 A 股特有的卖出单向征收，这是回测中必须考虑的，否则会严重高估收益。

### Q9: 数据量有多大？性能怎么样？

> 当前管理 5,684 只 A 股的数据，日 K 线约 1020 万条（跨度从 1991 年至今）。单次量化分析（300-1500 只股票、500+ 交易日、50 因子）耗时 30-90 秒，在后台线程池执行不影响 API 响应。K 线查询通过 (stock_id, date) 复合索引优化，单只股票 10 年数据查询在毫秒级。前端做了懒加载和分页优化——股票列表分页 50 条/页，搜索使用远程模式按需加载。

### Q10: 如果数据源全部挂了怎么办？

> 首先这种情况极少发生——6 个源来自不同的服务商（两个 Python 库、四个 HTTP API），同时宕机概率很低。但系统有降级策略：(1) FetchLog 记录失败信息，下次定时任务会重试；(2) K 线有缺口感知机制，只要数据源恢复就自动补全缺失数据；(3) 已有的历史数据仍可用，前端展示不受影响。最差情况下，用户在 Web 界面的「配置管理」中可以调整数据源优先级，手动触发重试。

### Q11: 安全方面做了什么？

> (1) JWT 认证 + bcrypt 密码哈希，24 小时 Token 过期；(2) RBAC 权限控制，8 个模块独立权限开关，admin 和 user 两种角色；(3) API Key 在 GET 响应中脱敏处理（只显示首尾各 4 位）；(4) 配置敏感信息（密码、Token）通过 `.env` 文件管理，不进入代码仓库；(5) 前端 401 拦截器自动登出过期会话。

### Q12: 你是怎么保证代码质量的？

> (1) 100 个集成测试覆盖数据源连通性、聚合器 fallback、数据库 CRUD、API 端点、端到端抓取流水线；(2) pytest 条件标记保证测试在不同环境（有网/无网、有 DB/无 DB）都能正常运行；(3) 9 个测试模块 + 2 个 E2E 模拟脚本提供了从单元到系统级的验证；(4) Pydantic Schema 在请求/响应两侧做数据验证；(5) AGENTS.md 文档持续记录架构决策和开发约定。

### Q13: 这个项目你一个人做的吗？花了多长时间？

> 是的，从架构设计到前后端实现到测试部署全部独立完成。（如果需要可以说明大概的时间跨度和开发节奏，这里建议根据实际情况作答。）

### Q14: 如果让你重新做一次，会有什么不同？

> 主要两点：(1) 前端会从一开始就做组件拆分 + TypeScript + Vue Router，而不是在一个 App.vue 里写完所有逻辑，这样后期维护和测试都更方便；(2) 量化引擎会考虑引入更成熟的框架（如 Qlib）作为底层，在其上做定制，而不是完全从零实现——虽然从零实现让我深入理解了每个细节，但工程上可以更高效。

### Q15: 项目中最有成就感的部分是什么？

> 多数据源聚合引擎。因为它真正解决了一个实际痛点——A 股免费数据源都不稳定，没有人可以保证 100% 可用。我通过适配器模式 + 优先级 fallback + 两阶段 K 线获取 + 缺口感知补全，让系统在各种故障场景下都能稳定工作。这个设计体现了我对系统可靠性的思考，不只是"能跑"，而是"稳定地跑"。
